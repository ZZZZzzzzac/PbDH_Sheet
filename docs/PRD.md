# PbDH Sheet Framework — PRD

版本 1.0，第一版。

---

## Problem Statement

Powered by Daggerheart（PbDH）TRPG 系统有多个衍生规则变体。每个规则的作者想做线上车卡器，但当前没有通用框架可用：

- 现有车卡器都是针对单一规则硬编码的 Web 应用，不同规则之间无法复用。
- 非程序员的 PbDH 作者缺乏技术能力搭建车卡器。
- 所有车卡器的核心能力（文本编辑、资源计数、选项选择、导入导出、打印）高度重合，重复造轮子。
- PbDH 规则的核心特色——"资源卡牌化"——没有通用的卡牌展示和管理系统。
- 作者定义的资源、规则、布局没有统一的格式，也无法校验结构正确性。

作者的需求是：给出规则、资源、页面布局，得到一个可用的车卡器。玩家需要建卡、编辑、管理资源、保存、打印、导入导出。

## Solution

做一个 **AI 协作式基础框架**：

- **作者侧**：定义系统包（规则文本 + 资源库 + 字段模块 + 页面布局 + 样式 + 检查规则）。作者自己写或让 AI 辅助生成系统包。框架提供预览页、校验器、面向 AI 的文档。
- **玩家侧**：导入系统包后得到车卡器。建卡、编辑、管理卡牌、导入导出、打印。

基础框架是所有车卡器的通用底座，系统包是插件。

**核心数据原则**：角色卡/资源库里的值默认就是字符串——如同线下纸笔，每个格子里写的是字符。基础框架不理解"等级""费用""伤害"的游戏语义，作者输入什么就显示什么。只有框架结构关键字段（ID、引用、模块类型等）才做严格校验。

**卡牌系统**：卡牌是 PbDH 的核心资源形态，不是普通选择文本的皮肤。卡牌支持纯文字版和图片版，有独立的展示和管理区域。卡牌系统的规则约束不由卡牌系统强制执行，而由只读检查模块报告。

以上决定不进入具体实现方案，只记录收敛后的需求边界。

## User Stories

### 作者侧

#### 系统包创建与管理

1. 作为一个 PbDH 作者，我想要用声明式系统包（文件、目录包或压缩包）定义规则、资源、布局和样式，以便我能直接编辑或让 AI 辅助生成。
2. 作为一个 PbDH 作者，我想要系统包内的所有数据键名使用中文，以便我能直接阅读、编辑原始数据文件。
3. 作为一个 PbDH 作者，我想要导入一个已有的系统包文件到框架中，以便我能在此基础上修改或参考。
4. 作为一个 PbDH 作者，我想要框架缓存当前系统包，以便我每次打开预览时不需要重新导入。
5. 作为一个 PbDH 作者，我想要框架只加载一个当前系统包，以便预览和编辑的上下文清晰不混淆。

#### 模块定义

6. 作为一个 PbDH 作者，我想要添加自由文本模块到角色卡页面，以便玩家能填写姓名等任意文本。
7. 作为一个 PbDH 作者，我想要添加长文本模块（文本框），以便玩家能填写大段描述或笔记，如背景、冒险笔记。
8. 作为一个 PbDH 作者，我想要添加可加减资源模块，以便玩家能管理生命、压力、恐惧点、限速、倒计时等有当前值/上限/梯级的资源量。
9. 作为一个 PbDH 作者，我想要添加勾选资源模块，以便玩家能用勾选框表达离散状态（如熟练度选择、已学会能力）。
10. 作为一个 PbDH 作者，我想要添加选择文本模块，以便玩家能从资源库中选择职业、武器、护甲、卡牌等条目。
11. 作为一个 PbDH 作者，我想要添加图片字段模块，以便玩家能上传头像/立绘或引用系统包内的图片。
12. 作为一个 PbDH 作者，我想要添加只读展示模块，以便在页面中展示规则速查、说明文、静态图片等固定内容。

#### 资源库定义

13. 作为一个 PbDH 作者，我想要定义多个资源库，以便管理职业、武器、护甲、卡牌、特长等不同类型的可选资源。
14. 作为一个 PbDH 作者，我想要每个资源库条目有必填字段（ID、名称、描述），以及可任意扩展的键值对，以便适应不同规则的字段需求。
15. 作为一个 PbDH 作者，我想要资源库里的所有值都默认是字符串，以便我可以写"等级X，X为当前等级一半"这样的非标准表述而不触发校验报错。
15. 作为一个 PbDH 作者，我想让资源库可以按库中条目的字段筛选/排序/搜索，并且同时可以应用多个条件。

#### 依赖逻辑

16. 作为一个 PbDH 作者，我想要定义可见性依赖：当玩家选择某个选项后，资源库相关条目或页面才显示。
17. 作为一个 PbDH 作者，我想要定义选项过滤依赖：当玩家选择职业 A 后，子职业资源库只显示此职业可用的子职业。
18. 作为一个 PbDH 作者，我想要定义字段自动填充：当玩家选择某武器后，自动填入名称、伤害、特性等文本。
19. 作为一个 PbDH 作者，我要求依赖逻辑只在声明式配置层面工作，不需要写脚本实现。

#### 检查数值

20. 作为一个 PbDH 作者，我想要定义只读检查规则（可以写检查脚本），在检查时一次性读取整张角色卡的数据，计算期望值，输出检查报告。
21. 作为一个 PbDH 作者，我要求检查脚本只能读取角色卡数据，不能修改任何模块状态。
22. 作为一个 PbDH 作者，我要求检查脚本可以输出检查项列表；每个检查项至少包含级别和文本说明，框架可选支持字段路径、错误码等机器可读信息。
23. 作为一个 PbDH 作者，我要求检查数值不自动写回字段，保持纸笔工具的自由度。
24. 作为一个 PbDH 作者，我希望能够定义导出/打印前强制检查，避免玩家带着错误数据输出。

#### 页面布局

25. 作为一个 PbDH 作者，我想要用 Flow Layout（页面 -> 分区 -> 行/列 -> 模块）声明式定义页面布局。
26. 作为一个 PbDH 作者，我想要定义多个页面，并指定页面顺序和显示条件。
27. 作为一个 PbDH 作者，我想要指定哪些页面可以打印。
28. 作为一个 PbDH 作者，我想要调整模块的标题、宽度、颜色、字体等样式属性。

#### 卡牌系统定义

29. 作为一个 PbDH 作者，我想要定义资源库中的条目以卡牌形态呈现给玩家。
30. 作为一个 PbDH 作者，我想要卡牌支持纯文字版和图片版两种形态。
31. 作为一个 PbDH 作者，我要求卡牌的最小必填字段为 ID、名称、描述，其他字段自由扩展。
32. 作为一个 PbDH 作者，我想要定义卡牌区域：配置（手牌）、宝库（牌组）、资源库（未获得）。
33. 作为一个 PbDH 作者，我要求卡牌的规则约束（如配置数量限制、等级要求）不由卡牌系统强制，而由检查数值报告。
34. 作为一个 PbDH 作者，我想要分离"资源库的浏览/筛选/排序"和"条目选择后的展示/使用形态"。

#### 车卡指引定义

35. 作为一个 PbDH 作者，我想要声明式定义车卡指引，以便玩家能按步骤完成建卡。
36. 作为一个 PbDH 作者，我想要为每个指引步骤编写说明文本，并引用目标页面、目标模块、资源库或卡牌区域，以便框架能把玩家带到正确位置。
37. 作为一个 PbDH 作者，我想要步骤能根据当前角色数据、已选资源或卡牌状态显示不同提示，以便指引适配不同职业、种族、社群或规则分支。
38. 作为一个 PbDH 作者，我想要步骤能触发框架允许的动作：跳转页面、聚焦/高亮模块、打开资源库选择、添加卡牌、通过既有依赖逻辑自动填充字段。
39. 作为一个 PbDH 作者，我想要步骤能声明简单完成条件和最终检查，以便玩家知道哪些步骤已完成、哪些仍需补齐。
40. 作为一个 PbDH 作者，我要求车卡指引不允许注入任意 UI 脚本或绕过框架动作直接修改角色数据。

#### 作者预览

41. 作为一个 PbDH 作者，我想要及时预览当前系统包生成的玩家侧车卡器效果，以便判断规则资源、布局、样式、依赖关系是否正确。
42. 作为一个 PbDH 作者，我希望预览反馈尽可能快，最好接近实时。
43. 作为一个 PbDH 作者，我想要在预览中看到系统包校验器的错误信息。

#### 系统包校验

44. 作为一个 PbDH 作者，我想要框架自动校验系统包的：必填结构字段缺失、系统包 ID/版本缺失、ID 重复、引用不存在、页面引用了不存在的模块、模块引用了不存在的资源库、依赖条件引用了不存在字段、自动填充目标不存在、车卡指引引用不存在的页面/模块/资源库/卡牌区域、卡牌缺 ID/名称/描述、中文键名拼写错误、图片引用不存在、检查脚本运行失败。
45. 作为一个 PbDH 作者，我想要校验器输出清楚的错误位置（文件/路径/字段）、错误原因和修复建议。
46. 作为一个 PbDH 作者，我想要校验器不把普通字段值强制校验成 int/float/bool 类型。
47. 作为一个 PbDH 作者，我需要校验器额外输出给 AI 看的 debug 日志（不面向作者展示）。

#### 文档与 AI 协作

48. 作为一个 PbDH 作者，我想要框架提供完整的作者文档：系统包格式说明、每种模块的字段说明、资源库/卡牌库格式说明、依赖规则说明、车卡指引说明、检查脚本说明、布局说明、样式说明。
49. 作为一个 PbDH 作者，我想要文档中有最小系统包示例和完整系统包示例。
50. 作为一个 PbDH 作者，我想要文档结构稳定、标题明确、示例完整、用语直接，便于 AI 读取和生成。
51. 作为一个 PbDH 作者，我想要框架提供面向 AI 的系统包生成提示词模板和工作流说明。
52. 作为一个 PbDH 作者，我想要校验器给出的错误信息也适合 AI 读取和自动修复。

## Implementation Decisions

### 架构定位

- 基础框架是一个固定 Web 应用，系统包是独立可分发数据包。
- 系统包格式为声明式文件、目录包或压缩包，不由作者写任意脚本（检查脚本例外，检查脚本只能读数据）。
- 第一版不做低代码可视化编辑器、不做拖拽布局编辑器。
- 第一版只加载一个当前系统包，不做多系统包管理器。
- 作者侧数据键名使用中文，作者可见格式和代码对象都直接用中文键名。

### 模块类型清单

第一版基础 Sheet Modules：

| 模块 | 说明 |
|------|------|
| 自由文本 | 单行文本输入，玩家自由填写 |
| 长文本 | 多行文本域，适合背景、笔记 |
| 可加减资源 | 当前值 + 上限 + 梯级 + 加减按钮 |
| 勾选资源 | 离散勾选框 |
| 选择文本 | 从资源库选择条目并填入 |
| 图片字段 | 上传或引用图片 |
| 只读展示 | 纯展示文本/图片，不可编辑 |

第一版框架运行时能力：

| 能力 | 说明 |
|------|------|
| 卡牌展示/管理 | 卡牌区域、卡牌操作 |
| Flow Layout 渲染 | Sheet Renderer 根据页面/分区/行/列声明安排 Sheet Modules |
| 检查报告 | 检查数值的输出展示 |
| 车卡指引 | 按作者声明的步骤引导玩家建卡 |

第一版不做：表格模块、富文本编辑器、时间轴、地图、公式字段、自定义交互组件。

### 依赖逻辑范围

第一版只支持三类：
1. **可见性依赖**：选择某选项后显示相关字段/页面。
2. **选项过滤**：选择某选项后过滤其他下拉框的可选项。
3. **自动填充**：选择条目后填入关联字段的文本。

不做：自动写回多字段、复杂公式系统、多步状态机、自动回滚升级。

### 检查数值

- 检查脚本只读角色卡全部数据（与导出 JSON 格式一致）。
- 检查脚本只能输出检查报告，不能修改任何模块状态，不能访问其他系统资源。
- 检查报告输出为检查项列表；每项至少包含 `level` 和 `text`，可选包含 `path` 和 `code`。
- 检查脚本在 Web Worker 中执行，输入为角色数据和资源库数据副本。
- 检查时机为手动检查和导出/打印前检查。

### 卡牌系统

- 卡牌区域：配置（手牌）、宝库（牌组）、资源库（未获得）。配置和宝库间可交换，资源库往配置/宝库单向流动。
- 卡牌数据必填：ID、名称、描述。其他字段自由扩展。
- 卡牌支持文字版（纯文本展示）和图片版（卡图展示）。
- 第一版卡牌交互承诺浏览、详情查看、加入、移动、排序、删除、翻面、横置/竖置、指示物数字加减或设置。
- 卡牌的规则约束（数量上限、等级要求、职业限制等）不进卡牌系统，由检查数值模块报告。
- 资源库的"浏览/筛选/排序"与条目被选后的"展示/使用形态"分离。

### 布局

- 第一版只做 Flow Layout：页面 -> 分区 -> 行/列 -> 模块。
- Flow Layout 兼容旧的 `section.modules` 简写；该简写继续使用自动响应式模块网格。
- Flow Layout 支持 `section.rows[].columns[].modules`，Author 可以声明行、列、列宽、最小列宽和模块摆放顺序。
- Flow Layout 支持页面、分区、行、列、模块摆放位的基础尺寸和盒模型样式，包括宽度、最小/最大宽高、间距、外边距、内边距、背景色、边框、圆角和对齐。
- Overlay Layout（底图叠加坐标定位）列为 future plan。
- 布局只影响模块位置和尺寸，不影响模块数据、依赖关系、检查规则。
- 第一版不单独拆 Layout Renderer；Sheet Renderer 负责 Flow Layout、页面结构渲染和调用 Module Registry。
- 只有当出现第二种实际布局 adapter（如 Overlay Layout）或 Sheet Renderer 接口明显变浅时，再重新评估是否拆出独立布局模块。

### 车卡指引

- 车卡指引由 System Package 声明，不由 Base Framework 硬编码具体步骤。
- 每个步骤可包含说明文本、目标页面/模块、目标 Resource Library、目标 Card 区域、完成条件和允许的框架动作。
- 指引动作只允许使用框架既有能力：页面导航、模块聚焦/高亮、打开资源选择、添加/移动卡牌、触发声明式自动填充、运行检查报告。
- Guide Runner 只维护当前步骤、完成状态和 UI 提示状态；它不是规则引擎，也不是任意脚本执行器。
- 复杂合法性判断仍由检查数值模块报告；车卡指引只做简单完成条件和流程提示。
- 玩家可以跳过、返回或重做步骤；指引不硬阻止玩家保留纸笔式自由填写。

### 数据存储

- 角色数据：本地浏览器存储，支持多角色存档。实现上大数据使用 IndexedDB，localStorage 只保存当前系统包、当前角色和少量 UI 状态。
- 角色数据包含：文本字段值、资源选择 ID、卡牌状态、系统包标识/版本、玩家上传的图片（仅头像/立绘）。
- 角色数据不包含：系统包完整资源库、系统包图片二进制（只保存引用）、云端数据。
- 系统包单独导入并缓存到本地，角色数据绑定系统包 ID/版本进行兼容检查。

### 导出与打印

- 第一版 HTML 导出为只读快照，不导出可继续编辑的完整 Sheet Tool。
- 第一版 PDF 导出通过浏览器打印/另存 PDF 实现，不内置前端 PDF 生成器。
- 导出/打印前运行检查脚本，发现 warning/error 时提示玩家，但不硬阻止继续导出或打印。
- 打印使用当前显示图；图片加载失败时保留文字版 fallback。

### 系统包校验器

校验规则分两类：
- **结构校验**（必须严格）：必填字段、系统包 ID/版本、ID 唯一性、引用完整性、模块类型匹配、页面-模块关联、资源库引用、依赖条件引用、自动填充目标、车卡指引步骤引用、卡牌 ID/名称/描述、中文键名拼写、图片引用存在性、检查脚本语法合法性和运行失败。
- **值校验**（不做强制）: 不校验字段值是否为 int/float/bool，不因非标准写法报错。

校验器输出分两层：面向作者的清洁错误信息（位置 + 原因 + 修复建议），以及面向 AI 的 debug 日志。

### 作者文档与 AI 协作

文档为第一版一等需求，不是附属物。包含：
- 系统包格式说明
- 最小/完整系统包示例
- 每种模块的字段说明
- 资源库/卡牌库格式说明
- 依赖规则说明
- 车卡指引说明
- 检查脚本说明
- Flow Layout 布局说明
- 样式可调整项说明
- 角色数据导入导出说明
- 校验器错误说明
- AI 生成系统包的提示词模板和协作工作流
- 常见错误与修复建议

文档必须偏 AI 友好：结构稳定、标题明确、字段用表格列出；每个核心概念尽量给最小 JSON 示例、反例、校验规则和生成步骤。

## Testing Decisions

本项目为绿地项目（从零开始），没有现有测试可参考。

### 测试原则

- 只测试外部行为，不测试内部实现细节。
- 基础框架的每个模块应可通过公开接口独立测试。
- 系统包校验器应有完整的错误/边界测试覆盖。

### 需要测试的模块

- 系统包校验器：测试所有结构校验规则对合法/非法输入的响应。
- 基础模块渲染：测试每种模块类型能正确渲染数据并响应玩家交互。
- 依赖逻辑引擎：测试可见性、选项过滤、自动填充的正确触发和边界情况。
- 检查数值执行环境：测试检查脚本的沙箱隔离、只读性、报告输出格式。
- 角色数据导入导出：测试导出完整性和导入兼容性检查。
- 卡牌区域操作：测试配置/宝库/资源库之间的卡牌移动、排序、翻面、横置和指示物数字变化。
- 车卡指引：测试步骤推进、跳过/返回、目标引用、高亮/导航效果、资源选择请求、允许的自动填充和最终检查触发。

### 不做测试的

- 具体规则的正确性（不是框架职责）。
- 特定样式/视觉还原度。
- 浏览器兼容性（第一版聚焦核心功能）。

## Out of Scope

以下明确列入第一版不做范围：

### 作者侧不做

- 低代码可视化编辑器（拖拽/点击/填表式）
- 拖拽布局设计器
- 图形化条件编辑器
- 图形化车卡指引编辑器
- 在线样式面板
- 作者账号/项目管理
- 在线发布平台（系统包市场）
- 多系统包管理器
- 打包整套源码给作者下载
- Overlay Layout（底图叠加定位）——列 future plan

### 玩家侧不做

- 骰子机器人导出（海豹骰子等）
- 悬浮笔记本
- 双页模式（列 future plan）
- GM 仪表盘作为内置特性
- 角色跨系统包转换/迁移
- 云同步/多设备同步
- 角色分组/权限/账号

### 卡牌不做

- 完整卡牌编辑器（列 future plan）
- 卡牌市场/分享
- 抽牌、弃牌、洗牌、费用支付等卡牌游戏机制
- 复杂卡牌多面状态（列 future plan）
- 多种指示物类型、实体指示物摆放和动画（列 future plan）
- 任意桌面沙盒
- 卡牌碰撞/堆叠规则
- 多人同步/实时协作

### 依赖逻辑不做

- 任意脚本修改角色卡状态
- 自动写回复杂数值计算
- 复杂状态机
- 多步回滚撤销
- 跨页面复杂联动

### 通用不做

- 允许作者写任意 JS 函数操作角色卡状态或 UI 行为
- 外部服务交互（骰子机器人、云端同步、在线卡库）
- 非规则相关的 UI 特效（动画、视觉主题、纸张做旧、仪表盘彩蛋）
- 自动从规则书生成系统包
