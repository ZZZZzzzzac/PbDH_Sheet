# PRD：Daggerheart 核心人物卡 System Package

## Problem Statement

现有 Daggerheart 车卡器以独立 HTML、CSS、全局 JavaScript 数据和命令式交互实现，无法直接复用 PbDH Sheet Framework 的 System Package、Character Data、Resource Library、Dependency Logic、Card Engine、Validation Check 与输出能力。Author 需要把核心人物卡重构为声明式 System Package，使 Player 能在 Base Framework 内创建、编辑、保存、恢复、检查和输出 Daggerheart 角色，而不迁移旧皮肤、自定义卡包和卡牌编辑器等包外功能。

## Solution

创建一个 `schemaVersion: 0.1.0` 的 Daggerheart System Package。迁移并正规化旧车卡器的核心规则数据，声明人物卡 Pages、Sheet Modules、Resource Libraries、Dependencies、统一 Card Table、Validation Checks 与基础语义化 HTML Layout。

第一版不追求旧页面的像素级定位。HTML 只按身份属性、资源 Counter、经历职业、装备物品、背景关系和卡牌桌面组织大致关系，Author 后续手工调整 CSS。生命、压力、希望、护甲、熟练度和金币使用 `countableResource`。职业与护甲 Picker 通过 `fillCountable` 设置 Counter 上限；派生阈值由 Player 填写并由 Validation Check 检查。所有卡牌资源通过一个支持 `资源库IDs` 的 Card Table 共用坐标系。

## User Stories

2. As a Player, I want to enter level, evasion and six traits, so that the core character statistics appear on the sheet.
3. As a Player, I want to record major and severe damage thresholds, so that I can resolve incoming damage during play.
4. As a Player, I want Validation Checks to compare my thresholds with armor base thresholds plus level, so that mistakes are reported without the framework changing my sheet.
5. As a Player, I want HP, Stress, Hope, Armor, Proficiency and Gold represented as counters, so that I can track current values compactly.
6. As a Player, I want to edit supported counter maxima, so that advancement and temporary rule changes remain possible.
7. As a Player, I want class selection to set the HP maximum from the class resource, so that the class's initial HP is applied consistently.
8. As a Player, I want armor selection to set the Armor counter maximum from the armor resource, so that armor capacity matches the selected armor.
9. As a Player, I want to choose one pure ancestry from a searchable Resource Library, so that its name and card become available without manual copying.
10. As a Player, I want to choose a community from a searchable Resource Library, so that its name and card become available without manual copying.
11. As a Player, I want class selection to fill class name, evasion, class features and default background/connection questions, so that common class information is transferred to the sheet.
12. As a Player, I want to open subclass selection manually with a class-specific default filter, so that I control when subclass selection occurs.
13. As a Player, I want class selection to apply suitable default filters to domain-card selection, so that relevant cards are easier to find.
14. As a Player, I want background and connection questions to remain editable after class defaults are filled, so that I can customize the prompts.
15. As a Player, I want separate editable answers for three background and three connection questions, so that I can record character history and party relationships.
16. As a Player, I want to record five experiences and their modifiers, so that experience bonuses are available during play.
17. As a Player, I want to choose primary, secondary and two backup weapons from Resource Libraries, so that equipment lookup is fast.
18. As a Player, I want weapon name, trait, range, damage, damage type, description, hand requirement and weapon category stored in separate readable fields, so that no runtime string formatting is required.
19. As a Player, I want to choose armor and fill its name, base thresholds, armor value and description, so that armor information is available on the sheet.
20. As a Player, I want five independent item slots, each with a Picker connected to the shared loot Library, so that items and consumables can be selected independently.

26. As a Player, I want the sheet, story page and card page included in appropriate output, so that the character can be printed or exported.
27. As a Player, I want Character Data import/export and local saves handled by the Base Framework, so that the System Package does not duplicate persistence logic.
28. As an Author, I want migrated resources to use stable unique IDs and normalized fields, so that references remain valid across package versions.

30. As an Author, I want a basic semantic HTML Layout without absolute positioning, so that I can refine the visual layout later.

32. As an Author, I want validation scripts tested against representative valid and invalid characters, so that rule warnings are trustworthy.

## Implementation Decisions

- Build one declarative Daggerheart System Package using schema version `0.1.0`.
- Use three Pages: core character sheet, story/relationships, and shared card table.
- Use semantic HTML Layout Templates and page-scoped CSS without absolute positioning in the initial implementation.
- Migrate old JavaScript datasets into JSON Resource Libraries with stable IDs; data may be renamed, split, merged or augmented to fit framework contracts.
- Provide Resource Libraries for ancestries, communities, classes, subclasses, primary weapons, secondary weapons, combined backup weapons, armor, loot, domain cards and beast forms.
- Remove the old hidden `ClassDomainTextbox`; class-domain data remains Author Data used by Dependencies and filters.
- Represent ordinary player-entered values as `freeText`/`longText`, the portrait as `imageField`, and resource tracks as `countableResource`.
- Keep weapon name, stat, range, damage, damage type and description as separate Sheet Values.
- Implement five item text modules and five Resource Pickers sharing the loot Library.
- Use `fillText` for resource-to-text mapping and `fillCountable` for class HP maximum and armor maximum.
- Keep subclass selection manual; class selection only changes its default filter.
- Use one Card Table whose `资源库IDs` includes every card-producing Library. Each contributing Picker targets this Card Table.
- Ensure every entry consumed as a Card Definition has non-empty configured name and description fields.
- Use Validation Checks for armor threshold consistency and other read-only gameplay diagnostics. Validation never mutates Character Data.
- Keep Resource Picker selections transient; only Dependency-written Sheet Values and Card Instances persist.
- Do not reproduce arbitrary scripts, runtime DOM behavior or old localStorage keys in the System Package.

## Testing Decisions

- Tests assert external package behavior and diagnostics, not internal implementation structure.
- Validate the complete generated package through the same Loader/Validator pipeline used by Author Preview.
- Add focused tests for migrated Resource Library normalization, stable/unique IDs and required Card Definition fields.
- Test representative Picker Dependencies: class text fill, class HP maximum, armor text/max fill, subclass/domain filters, equipment fields and five item slots.
- Test that all configured card-producing Pickers can create cards on the shared multi-Library Card Table.
- Test Validation Check behavior with valid thresholds, incorrect thresholds, missing/non-numeric values and relevant boundary cases.
- Use existing dependency-engine, package-validator, Card Table rendering and canonical demo Validation Check tests as prior art.
- Perform an Author Preview smoke test covering page rendering, one selection from each major Library, Character Save reload, shared card interactions and output mode.

## Out of Scope

- Pixel-perfect recreation or final manual CSS positioning.
- Mixed-ancestry generation.
- Automatic subclass modal opening.
- Automatic calculation or mutation of damage thresholds.
- Tri-state checkbox/slot presentation.
- Runtime custom card-pack upload.
- Player-authored Card Definition editor.
- Legacy skin selection and Skin API.
- Enemy/scene card generators.
- Migrating the separate `nmxt` application.
